# FP Plugin Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a Claude Code plugin (`fp`) that applies first-principles thinking to software design review, architecture review, planning, and code review.

**Architecture:** Four separate skills (design, architecture, plan, code) share a common five-phase reasoning framework, principle catalog, and output templates via shared reference files. The plugin is structured for Claude marketplace distribution.

**Tech Stack:** Markdown (SKILL.md files), JSON (plugin manifest). No executable code — this is a pure content plugin.

### Task 2: Shared Framework Reference

**Files:**
- Create: `skills/shared/framework.md`

- [ ] **Step 1: Write the five-phase framework**

Create `skills/shared/framework.md` containing the complete five-phase reasoning framework that all four skills reference. This is the core engine of the plugin.

Content structure:
1. **Overview** — Brief description: a five-phase reasoning engine, not a checklist
2. **Phase 1: Decompose** — Instructions for stripping the problem to fundamental truths
   - What problem is actually being solved? (Not the solution — the problem)
   - Who are the actors? What are their real needs?
   - What are the hard, irreducible constraints? (Physics, business rules, regulatory, resource)
   - What would a solution look like with zero legacy, zero convention, zero bias?
3. **Phase 2: Surface Assumptions** — Instructions for identifying what's taken for granted
   - What technology choices were made by default rather than by reasoning?
   - What patterns are followed because "that's how it's done" vs. because they're the right fit?
   - What requirements are actually requirements vs. inherited preferences?
   - What's the cost of each assumption if it turns out to be wrong?
4. **Phase 3: Reason Upward** — Instructions for building from fundamentals toward the solution shape
   - What are the natural boundaries in this problem?
   - What must change together? What can change independently?
   - Where does data naturally live? How does it naturally flow?
   - What are the simplest structures that satisfy the real constraints?
5. **Phase 4: Anchor Against Principles** — Instructions for cross-checking against the principle catalog
   - Reference: `[principle-catalog.md](principle-catalog.md)`
   - Select principles based on relevance to the current mode and findings, not mechanically
   - For each relevant principle, apply its Key Question to the subject
6. **Phase 5: Produce Assessment** — Instructions for generating layered output
   - Direct findings (what's aligned, what's not, and why)
   - Severity classification (critical / moderate / minor)
   - Socratic questions for each finding
   - Concrete recommendations traced to principles or fundamental truths
   - Reference: `[output-templates.md](output-templates.md)` for the appropriate template
7. **Mode Priority and Cross-Skill Awareness** — Each skill should know the full set of modes and their priority order (Design > Architecture > Planning > Code Review). Include instructions for:
   - If the input seems better suited to a different mode, suggest switching (e.g., "This looks more like an architecture concern. Would you like to use `/fp:architecture` instead?")
   - If the user invokes a skill with no input and no codebase context, offer all four modes with brief descriptions rather than just asking for mode-specific input
   - If the input is ambiguous (could be design or architecture), state which mode you're using and offer to switch
8. **Depth Scaling** — How to adapt depth based on complexity
   - Simple (single component): Brief decomposition, 3-5 findings
   - Moderate (multi-component): Full decomposition, 5-10 findings
   - Complex (system-level): Deep decomposition, comprehensive findings with severity ratings
   - Natural language mapping: "light/quick/brief" → Simple, "thorough/detailed" → Moderate, "deep/comprehensive/exhaustive" → Complex

All instructions written in imperative form (telling the AI what to do).

3. **Planning Report** — Used by Planning mode
   - Header: title
   - The Problem (Decomposed)
   - Constraints (Irreducible)
   - Assumptions (To Be Validated)
   - Key Decisions (ordered by irreversibility), each with: what it determines, reasoning, options with trade-offs, reversibility rating

**Frontmatter:**
```yaml

name: design
description: >
  Use when reviewing a software design for first-principles alignment.
  Triggers on: "review this design", "evaluate this design", "design review",
  "is this design sound", or when a user shares a design doc/spec for feedback.
argument-hint: "[target file or description]"
---
```

**Body (imperative instructions for Claude):**

1. **Mode identification** — You are performing a First Principles Design Review. Your goal is to evaluate whether a software design follows from the actual problem's fundamentals or from inherited assumptions and conventions.

2. **Input gathering** —
   - If `$ARGUMENTS` contains a file path, read it
   - If `$ARGUMENTS` contains a description, use it as context
   - If no arguments, ask the user: "What design would you like me to review? Share a file path, paste the design, or describe it."
   - Also explore the codebase for related files (specs, existing implementation) using Glob/Grep/Read
   - Ask at most 2 targeted clarifying questions if the design context is unclear

3. **Execute the five-phase framework** — Reference: `[framework](../shared/framework.md)`
   - Phase emphasis: Heavy on Phase 1 (Decompose) and Phase 2 (Surface Assumptions). Phase 3 produces an "ideal shape" to compare against the proposed design. Phase 4 anchors against: Separation of Concerns, Cohesion, Information Hiding, Simplicity, Single Source of Truth (draw others as relevant).

4. **Output** — Use the Assessment Report template. Reference: `[output-templates](../shared/output-templates.md)`
   - Output focus: Gap analysis between the proposed design and what first-principles reasoning suggests
   - Highlight where the design is assumption-driven vs. fundamentals-driven

5. **Edge cases** —
   - Empty invocation with no codebase context: Offer all four modes with brief descriptions, highlighting that design review is recommended as a starting point
   - Empty invocation with codebase context: Ask what design to review, offer examples from the codebase
   - Input looks like code not design: Suggest `/fp:code` instead, proceed if user confirms
   - Input looks like architecture (service boundaries, deployment): Suggest `/fp:architecture` instead, state why, proceed if user confirms
   - Input too large: Focus on the most critical components, state what was analyzed and what was skipped

**Body (imperative instructions for Claude):**

1. **Mode identification** — You are performing First Principles Planning. Your goal is to build a principled foundation for a new project or feature by decomposing the problem before proposing any solution.

2. **Input gathering** —
   - If `$ARGUMENTS` contains a description, use it as the problem statement
   - If in an existing codebase, explore it for context (what exists, what patterns are established)
   - Ask at most 3 targeted questions to understand: the real problem, the users/actors, the hard constraints
   - Do NOT ask about technology choices or solution approaches — those come from reasoning, not requirements

3. **Execute the five-phase framework** — Reference: `[framework](../shared/framework.md)`
   - Phase emphasis: Phase 1 is the star — deep decomposition before any solution is discussed. Phase 2 challenges premature solution assumptions ("I was thinking we'd use React and PostgreSQL" — why?). Phase 3 builds a skeleton design from fundamentals. Phase 4 validates the skeleton. Phase 5 is forward-looking: decisions with reasoning, open questions, suggested decision order (most irreversible first).

4. **Output** — Use the Planning Report template. Reference: `[output-templates](../shared/output-templates.md)`
   - Output focus: A principled foundation, not a complete design
   - Key decisions ordered by irreversibility (most costly to change first)
   - Each decision includes 2-3 options with trade-offs
   - Open questions should be genuine unknowns, not rhetorical

5. **Edge cases** —
   - Empty invocation with no context: Offer all four modes with brief descriptions
   - Vague problem statement: Ask clarifying questions (this mode allows up to 3)
   - Problem is too large: Help decompose into sub-problems, then plan the first one
   - Existing system extension: Acknowledge existing constraints as irreducible, focus planning on the new parts
   - User seems to want a review of something existing: Suggest `/fp:design` or `/fp:architecture`, state why

**Frontmatter:**

  "first principles code review", or when the user wants deeper-than-surface

**Body (imperative instructions for Claude):**

1. **Mode identification** — You are performing a First Principles Code Review. Your goal is to evaluate whether code structure and logic follow from what the code actually needs to do, or from assumptions, convention, and accidental complexity. This is NOT a style review — focus on structural and logical issues.

2. **Input gathering** —
   - If `$ARGUMENTS` contains a file path, read it
   - If `$ARGUMENTS` contains a PR number, fetch the diff using `gh pr diff`
   - If `$ARGUMENTS` is a description, use Glob/Grep to find relevant files
   - If no arguments, ask: "What code would you like me to review? Provide a file path, PR number, or describe what to look at."
   - Read surrounding context (imports, callers, tests) to understand the code's role

3. **Execute the five-phase framework** — Reference: `[framework](../shared/framework.md)`
   - Phase emphasis: Phase 1 infers the problem from the code (what is this code trying to do? what are its real responsibilities?). Phase 2 identifies code-level assumptions (magic numbers, implicit contracts, assumed execution order, hidden coupling). Phase 3 asks "given what this code needs to do, what's the simplest correct structure?" Phase 4 anchors against: Single Responsibility, Information Hiding, Coupling, Fail-fast, Simplicity.

4. **Output** — Use the Code Review Report template. Reference: `[output-templates](../shared/output-templates.md)`
   - Output focus: Concrete code-level findings with file:line references
   - Each finding includes a code snippet where helpful
   - Not style nits — structural and logical issues that first-principles reasoning reveals
   - Recommendations include specific file:location references

5. **Edge cases** —
   - Empty invocation with no context: Offer all four modes with brief descriptions
   - Empty invocation with codebase: Ask what code to review, suggest recently changed files
   - Massive codebase with no target: Ask the user to narrow scope
   - Input looks like a design doc or architecture description: Suggest `/fp:design` or `/fp:architecture`, state why, proceed if user confirms
   - Generated code or config files: Note that generated code has different first-principles considerations (the generator is the real subject)
   - Test files: Review test design through first principles (is it testing behavior or implementation?)
